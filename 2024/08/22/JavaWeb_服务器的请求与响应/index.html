服务器的请求与响应

Tomcat&Servelt容器

首先，Tomcat是一个后端服务器软件，其次，它被称为servelt容器，但为什么？

在初步了解Tomcat后，我尝试将一个后端程序部署到Tomcat中运行，结果发现在浏览器中无法访问，问题就处在这个servelt容器上：Tomcat识别的后端资源是有要求的——需实现servelt接口，Tomcat作为servelt容器，实质上就是管理servlet的生命周期，由于demo程序没有实现这个接口，因此无法被Tomcat识别

什么Servlet

一个运行在服务器端的 Java 类，选择实现了部分接口，用于处理客户端的请求并生成响应结果

DispatcherServlet

Spring框架提供了一个组件，使Spring程序可以部署到Tomcat中，这个组件就是DispatcherServlet，它实现了servelt接口

DispatcherServlet是Spring框架中的一个核心组件，它是 Spring MVC（Model-View-Controller）的前端控制器（Front Controller）。DispatcherServlet 负责将请求路由到应用程序中的适当处理器（handler）。它在 Spring MVC 应用程序中扮演着至关重要的角色，因为它是处理所有HTTP请求的中心点。

DispatcherServlet组件实现了servlet接口，可以被Tomcat识别与管理，因此作为中间组件，将请求转发到对应的控制器（处理http请求的组件），在控制器处理完成后，需要返回处理结果时，也会通过DispatcherServlet组件响应浏览器。 因此勾选了Web依赖的SpringBoot项目可以被Tomcat识别的原因就在于，项目包含了这个组件

请求响应

为实现接收请求，响应数据功能，servlet提供了两个接口

请求(HttpServletRequest):获取请求数据
响应(HttpServletResponse):设置响应数据

题外话：两个架构

应用的运作离不开请求与响应，而这两种架构对请求响应的依赖程度不同

  • BS架构:Browser/Server，浏览器/服务器架构模式。客户端只需要浏览器，应用程序的逻辑和数据都存储在服务端。

        优点：方便使用

        缺点：所有服务都依赖服务器，应用速度跟网速挂钩

  • CS架构:Client/Server，客户端/服务器架构模式。

        优点：由于可以在本地运行程序，应用运行更快

        缺点：不同系统需要不同的软件版本，开发/维护麻烦

HttpServletRequest是什么

Java Servlet API 的一部分，它代表了一个客户端发送到服务器的请求。这个对象包含了请求的信息，如参数、头信息、路径信息等

形参使用HttpServletRequest对象接收请求，通过getParameter方法（HttpServletRequest 对象的一个方法）解析对象中指定参数的字符串值（因此返回值为String），如果需要得到其他类型数据需手动处理数据类型（如Integer.parseInt(ageStr)）

利用Spring

Spring可以将请求参数自动分配到同名形参

参数名与形参变量名相同，定义形参即可接收参数

效果同上

两种用法区别在于，直接使用形参可以利用Spring的类型转换机制，而HttpServletRequest需要手动处理类型转换（getParameter）。

当参数名与形参变量名不同时，形参无法接受传参，也不会报错。

如果希望形参能接收不同名的与请求参数，需使用RequestParam注解指定请求参数的名称

@RequestParam 是 Spring MVC 中的一个注解，用于将 Web 请求中的参数（例如，来自 GET 或 POST 请求的查询参数或表单字段）绑定到控制器方法的参数上。使用 @RequestParam 可以很容易地从 HTTP 请求中获取参数值，并将其传递给控制器方法。

@RequestParam中的required属性默认为true，代表该请求参数必须传递，如果不传递将报错。如果该参数是可选的，可以将required属性设置为false。

(@RequestParam(name = “name”, required = false) String bushinam

这是什么

一种编码格式，通常用于表单提交数据到服务器。这种格式的数据被编码为键值对，键（key）和值（value）之间用等号（=）连接，不同的键值对之间用和号（&）分隔。即在我们编辑完成后，请求体会以{“key”=“val”}的形式传输。

需添加注解RequestParam

为什么集合形参需要使用注解？

在Spring MVC中，当你想要将一个HTTP请求的参数列表（例如，一个查询字符串中的多个值）绑定到控制器方法的参数时，使用@RequestParam注解是必须的。如果省略@RequestParam注解，Spring MVC将不会知道形参应该从请求参数中获取值，因此无法正确地将请求参数绑定到方法参数。

对于简单类型和数组，@RequestParam注解不是必须的，Spring MVC可以默认处理这些类型的参数。Spring会尝试自动将请求参数绑定到这个数组参数，而不需要显式地使用@RequestParam注解。这是因为Spring MVC提供了对简单类型的自动绑定支持，包括数组类型。

例如，如果请求的URL是http://example.com?param=value1&param=value2，Spring MVC会自动将value1和value2绑定到String[] param。

RequestParam与RequestBody

@RequestParam 和 @RequestBody 都是 Spring MVC 中用于处理 HTTP 请求的注解，但它们的用途和工作方式有所不同：

  1. @RequestParam：
  • 用于从客户端请求的查询参数（即 URL 中的参数）中获取值。
  • 可以应用于方法参数，将请求中的查询参数映射到方法的参数上。
  • 适用于 GET 请求或 POST 请求中的查询字符串参数。
  1. @RequestBody：
  • 用于将 HTTP 请求的正文（body）映射到方法的参数中。
  • 通常用于处理 POST、PUT、PATCH 等请求，这些请求的正文中包含了要发送给服务器的数据，通常是 JSON 或 XML 格式。
  • 需要在请求头中设置 Content-Type，例如 application/json，以便 Spring 能够正确解析请求体。
路径参数

使用PathVariable将请求的 URL 路径中的一部分绑定到控制器处理方法的参数上。这种方式允许你根据 URL 路径的不同部分来处理不同的请求。

也可以有多个路径参数，只需为所有路径参数添加注解即可

响应

后端将数据返回前端，是响应

ResponseBody注解

将方法的返回值作为响应，当返回值为实体对象/集合时，会转为json格式响应

上文请求处理类的注解RestController其实主要是两个注解的组合（其他为原注解）

  • Controller：标记一个类作为Spring MVC的控制器

  • ResponseBody：指示方法的返回值应该直接作为HTTP响应的正文返回，不添加这个注解时，处理方法的返回值会被尝试渲染为一个视图并返回

结合前后端完成一个小demo

前端：使用Vue管理页面元素，并运用Axios异步请求到数据用于动态渲染

后端：接收由前端发来的请求，对目标文件中的数据进行加工并响应（需统一响应结果）

因为只是一个小demo，并没有将前后端项目部署到不同的服务器，直接将前端页面存于SpringBoot项目中

项目创建步骤如下

  • 在pom.xml文件中引入dom4j的依赖，用于解析XML文件

  • 引入解析XML的工具类XMLParserUtils、对应的实体类Emp、XML文件 emp.xml

  • 引入一个静态页面文件，放在resources下的static目录下

  • 编写Controller程序，处理请求，响应数据

一、

pom.xml是maven的核心文件，提供依赖管理功能，写入dom4j的位置可以自动为项目添加这个依赖

dom4j是一个开源的库，用于处理 XML 和 HTML 文件，目的是将xml文件处理为多个便于操作的对象

二、

XMLParserUtils工具类用于解析xml文件，这个xml文件包含4个员工的信息，匹配出所有员工信息并纳入集合，处理成一个由员工类组成的集合

Emp实体类代表员工，里面的属性都跟员工有关

emp.xml是包含员工数据的文件

三、

将前端页面存在了SpringBoot项目中，需存放在以下三个地方之一：

  • classpath:/static

  • classpath:/public

  • classpath:/resources

classpath指类目录，在maven项目中resources就是类目录，底下已经创建好了static文件夹，将前端页面存在里面即可

四、

后端程序，是一个请求处理类，用于处理前端发送的请求，处理方法中将处理员工数据并包装为统一形式（一个Result对象）响应
