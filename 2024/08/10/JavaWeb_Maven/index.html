Maven基础

介绍

一款管理和构建Java项目的工具

功能

maven有三个作用

  • 依赖管理

  • 统一项目结构

  • 项目构建

依赖管理

无需手动（上网）导入依赖。

依赖是指项目中需要用到的外部库或模块,它们提供了一些功能,可以被项目直接使用,而不需要自己重新实现。

在Java项目中,依赖通常是以jar包的形式提供的。开发者可以将需要的jar包添加到项目的依赖库中,通过依赖管理工具(如Maven、Gradle)来自动下载和管理这些jar包。maven使用pom.xml文件来声明项目的依赖,并提供了丰富的依赖管理功能,如依赖传递、依赖范围、依赖冲突解决等。

统一项目结构

使用maven构建的java项目可以在不同的开发工具中打开，具有清晰的项目结构。

使用maven项目构建工具，可以让项目有统一的目录结构：

  • main：实际项目资源

        java：源代码目录

        resourse：配置文件目录

  • test 测试项目资源

        java

        resource

  • pom.xml：项目配置文件

项目构建

提供自动化项目构建方式

项目构建流程：

  • 编译

  • 测试

  • 打包

  • 发布

maven工具为这些步骤提供了运行指令，而且是标准跨平台的

Maven模型

模型：软件设计中的类、数据结构或算法的抽象表示。可以理解为对数据的处理办法。

maven模型主要包含以下三个模块：

构建生命周期/阶段

使用多个插件完成项目构建

项目对象模型（POM）

使用一段信息描述整个maven工程（坐标），通过坐标可以定位到工程

依赖管理模型

项目所需依赖，引入依赖坐标

maven会自动从仓库查找并导入依赖，仓库有三种

  • 本地仓库：主机上的目录

  • 中央仓库：全球唯一的仓库，包含几乎所有jar包，由Maven团队维护

  • 远程仓库：公司搭建的私有仓库

由于中央仓库在国外，访问速度有限，所以使用远程仓库充当缓存，maven查找依赖时，先从本地仓库开始，没有再访问远程仓库，远程仓库没有则从中央仓库导入到远程仓库，此时这个依赖就可以直接从远程仓库中获取了。

因此在在团队开发时，由于远程仓库的存在，只要有人导入过某个依赖，所有人均可以很快的获取它。

安装&配置

安装

[Maven官网](Maven – Download Apache Maven)

下载并解压

配置

本地仓库

<localRepository>path</localRepository>

镜像

环境变量

添加环境变量MAVEN_HOME，配置为maven的路径。

向Path添加maven的bin目录路径，用于全局使用指令

%MAVEN_HOME%\bin

可以在命令行通过指令检查是否配置成功

返回maven相关信息则表明配置完成

maven依赖jdk运行，所以电脑中需要提供版本合适的jdk

JDK（Java Development Kit）是Java开发工具包，它是Java开发环境的核心组件，包含了Java程序开发和运行所需的所有工具。JDK通常包括以下几个部分：

  1. Java编译器：用于将Java源代码编译成字节码。
  2. Java虚拟机（JVM）：用于执行编译后的Java字节码。
  3. Java类库：提供了Java开发所需的标准API。
  4. 各种工具：如javac、java、javadoc、jar等，用于程序的编译、运行、打包和生成文档等。

目录结构

bin

可执行文件，用于存放maven指令

conf

配置文件，如项目依赖，

lib

maven的依赖（基于java开发，也使用了很多jar包）

Maven的集成

在idea中集成maven

对于单一工程的配置

构建系统：Maven

设置——构建、执行、部署——构建工具——Maven：修改Maven主路径到Maven目录，用户设置文件重写为Maven的配置文件settings.xml

编译器——java编译器：版本与Maven一致

这样配置只能在这一个工程中使用，如果希望只配置一次，可以配置Maven环境。

Maven环境配制

回到Idea的开始页面：

  • 项目

  • 自定义 👈chose 所有设置

  • 插件

  • 学习

然后就跟项目配置步骤一样了

创建maven项目

项目配置选项：

  • 组 ID

  • 工作 ID（与项目名一致）

你问我为什么要写这些，因为这组成了maven的坐标

Maven坐标

资源的唯一标识，通过坐标可以唯一定位资源位置

使用坐标来定义项目或引入项目中需要的依赖

maven由下面三个部分组成

  • 组 ID

  • 项目 ID（与项目名一致）

  • 项目版本

配置文件中的其他信息

还有关于jdk版本，字符集

  • 当前使用jdk版本

  • 打包使用jdk版本

  • 项目源代码编码格式

导入maven工程

点击idea右边的m，添加所需项目的pom.xml即可导入工程

依赖管理

配置依赖

这里的依赖就是项目运行所需的jar包

配置:

  1. 在 pom.xml中编写<dependencies>标签
  2. 在<dependencies>标签中 使用<dependency>引入坐标
  3. 定义坐标的 groupld，artifactld，version
  4. 4.点击刷新按钮，引入最新加入的坐标

例：导入Logback（日志处理）依赖，scope表示依赖的范围

依赖传递

依赖具有传递性，即某项目依赖的依赖同样是这个项目的依赖

直接依赖：直接依赖:在当前项目中通过依赖配置建立的依赖关系
间接依赖：被依赖的资源如果依赖其他资源，当前项目间接依赖其他资源

idea提供了项目依赖关系的可视化工具

排除依赖

某项目不需要依赖的依赖，可以主动断开依赖

要排除依赖的信息添加到dependence内，用于排除这个依赖的依赖

需要指定组id(groupId)和工作id(artifactId)，不用指定版本

依赖范围

依赖默认可以在任何地方使用，也可以通过scope标签设置作用范围

  • 主程序范围有效。(main文件夹范围内)

  • 测试程序范围有效。(test文件夹范围内)

  • 是否参与打包运行。(package指令范围内)

scope值 主程序 测试程序 打包 示例
compile（默认） √ √ √ log4j
test - √ - junit
provided √ √ - servlet-aoi
runtime - √ √ jbdc驱动

生命周期

maven有三套独立的生命周期

  • clean：清理工作

  • default：核心工作

  • site：生成报告，发布站点等

每一套生命周期中有多个阶段，这里主要了解clean与default中的一些重要阶段

clean

clean（清理）：移除上一次构建生成的文件

default

compile（编译）：编译项目源码

test（测试）：使用合适的单元测试框架运行测试（如junit）

package（打包）：将编译后的文件打包

install（安装）：安装项目到本地仓库

同一套生命周期内阶段的运行具有先后顺序，后面阶段的运行需要依靠其前面的阶段。因此当运行install时，会自动运行compile、test、package阶段

生命周期、阶段都是抽象概念，不执行具体操作，而是由其绑定的插件完成

maven本质：插件执行框架

junit是一个流行的Java测试框架，用于编写和运行单元测试

  • junit
