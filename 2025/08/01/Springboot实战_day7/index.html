缓存菜品

数据库操作l很影响整个业务流程的速度，点外卖时不断切换分类，会导致多次查询重复数据，一不小心就卡住了

解决方案：用redis做缓存（按菜品类别）

DIshServiceImpl

先查询redis中是否有缓存数据，没有再去数据库查，最后缓存查到的数据

使用了缓存层，肯定会出现的问题是如何同步缓存与数据库数据。本项目的解决办法是在修改数据库时，清除受影响的缓存，也就是在增删改菜品时，要清除redis数据

共性方法，让我想到了AOP，能不能在方法上增强清除缓存操作呢？

使用Annotation，在需要处理的方法上添加注解，传需处理的pattern（模式，匹配key），不就可以了吗

抱着用用新东西的态度，我试着写了一下

一通操作下来，我开始往需要增强的方法上加注解

然后我就发现问题了，有些pattern是通过实参获取到的，我如何将实参内容传给注解呢

就像下面这个方法，修改单个菜品时，只用删除对应菜品类别的缓存数据即可，而菜品类别要通过dishDTO获取到

问了问ai，确实有一种让注解获取参数的办法：SpEL，有点复杂，而且我原本设计的是直接传入pattern，而不是拿个参数再在切面方法中拼出pattern，怪不得老师选择直接写成方法。再仔细想想，缓存层能跟业务分离吗？感觉相较于日志，缓存还是挺“业务”的，所以从设计上，这样去写也不太占优

Spring Expression Language (SpEL)

Spring表达式语言，用 字符串表达式 在运行时读取、计算对象属性或调用方法。即让注解或配置能动态读取对象值。

还是老实的写成方法吧

这样行吗

NO，我本来以为delete操作传是可以使用通配符的，仔细一看，只有查询操作能用

复习一下：通用命令

keys [pattern]：返回符合给定模式的key

exist [key]：检查给定key是否存在

type [key]：返回给定key所存储值的类型

del [key1] [key2]……：当给定key存在时，删除key

这样就可以了

要修改的方法有：

新增菜品：删除该类缓存

批量删除菜品：删除全部缓存

修改菜品：删除该类缓存

很奇怪，老师把清除缓存的操作直接写在了controller层，我不太理解

SpringCache

Spring提供的框架，实现了基于注解的缓存功能

我理解了，因为好删……

SpringCache提供了一层抽象，底层可以切换不同的缓存实现

  • EHCache

  • Caffeine

  • Redis

切换方式就是在pom里面配置了哪种缓存，SpringCache就用哪种

pom里配置的是java中的Redis客户端

Redis_Spring_Data

引入依赖

常用注解

  • EnableCaching：开启缓存注解功能——可使用注解配置缓存，加启动类上

  • CachePut：将方法的返回值加入缓存

  • Cacheable：方法执行前查询缓存中是否有数据，有则返回，没有则调用方法并更新缓存

  • CacheEvict：移除缓存中的数据

练习

打开专门的练习项目，创建数据库spring-cache-demo，运行项目里的sql创建user表

首先！在启动类上添加注解EnableCaching，这样才能使用注解操作缓存

共有四个要改造的方法

  • save

  • getById

  • deleteById

  • deleteAll

CachePut

将方法的返回值加入缓存

来改方法save：向数据库表中插入一条数据，观察到实际插入的数据就是数据库回填了id的user，因此选择CachePut注解：将方法的返回值加入缓存，填写了两个参数

  • cacheNames = “userCache”

  • key = “#user.id”

它们共同组成Redis中的key：userCache::user.id

补充：redis中的树形结构

使用 ‘:’ 构建树形结构

a : b : c，在redis中就会形成树形结构，而上面的userCache::user对应的结构就是

userCache-Empty-user.id

是不是见到一个眼熟的东西？SpEL，没想到这里还能用到

@CachePut(cacheNames = "userCache",key = "#user.id")

由于想要根据用户id动态设置redis中的key，所以使用了它，除了#user.id，还有很多写法

它们的作用是一样的，至少在这个方法上。当然，最见名知意的就是第一种

Cacheable

方法执行前查询缓存中是否有数据，有则返回，没有则调用方法并更新缓存

正好对应getById这种查询操作

@Cacheable的底层实现：动态代理

CacheEvict

移除缓存中的数据

deletyById：删除一个

deletyAll：删除全部

这个时候怎么办呢，key只能指定一个值。需要使用另一个属性allEntries

此时会删除userCache下的所有数据

插一句：git的使用

我之前完成了部分setmeal接口，但是由于整个功能运行需要其他接口，我不好测试，于是我将这部分可能有问题的代码提交到了分支上

new branch->check out这个新分支，这时项目就切换到新分支上了

今天完成了这部分的功能，测试没问题后，我想合并到主干上

先切到主干，在上方（idea）找到git-merge，然后可能会有多个可选项

可合并来源

  • master 本地 master 当前分支

  • origin/master 远端 master 远端有更新，Git 检测到可以合并

  • origin/setmealApi 远端 setmealApi 远端同名分支，可能是你推送过的

  • setmealApi 本地setmeal分支

选最后一个

合并方式

  • Create a merge commit even if it can be fast-forwarded--no-ff 强制生成一条合并提交（保留分叉历史） git merge --no-ff feature/xxx

  • Merge only if it can be fast-forwarded--ff-only 只能“快进”合并；若存在分叉，则拒绝合并 git merge --ff-only feature/xxx

  • Create a single commit for all merged changes--squash 压缩所有改动为一条提交，不产生合并提交 git merge --squash feature/xxx

  • Specify a message for the merge commit-m 给合并提交自定义日志 git merge -m "合并 xxx"

  • Do not commit the merge result--no-commit 只把改动放进暂存区，不自动提交，方便再改 git merge --no-commit feature/xxx

  • Bypass pre-merge / commit hooks--no-verify 跳过钩子脚本（如代码检查） git merge --no-verify feature/xxx

  • Allow merging histories without common ancestor--allow-unrelated-histories 允许合并两个互不相关的历史（常见新仓库） git merge --allow-unrelated-histories

保留完整历史：勾选 –no-ff （默认）。
线性历史：选 –ff-only 或 –squash 。
手动改合并结果：勾选 –no-commit 。

总结

今天学了SpringCache，是十分有用的工具，我开始想到问题都被SpringCache解决了，然后补上了套餐相关接口，并完成了购物车功能

复习了

  • redis的使用

  • redis_spring_data：java的redis客户端，.delete()不能直接使用模式串，要通过.keys(模式串)获取匹配项的集合，使用集合删除对应项

  • 线程变量：BaseContext.getCurrentId()

  • AOP：annotation

学习了

  • redis：树形结构 ‘:’

  • SpringCache：Spring提供的工具，可用注解操作缓存，四个常用注解

    @EnableCaching：添加在启动类上，开启注解操作缓存

    @Cacheput：将返回值写入缓存

    @Cacheable：查询缓存，有则直接返回，无则执行方法，将返回值载入缓存

    @CacheEvict：删除缓存数据，需要删除某目录下全部数据时，使用allEntries = ‘true’

了解了

  • SpEL：Spring Expresion Language，spring表达式语言，可在注解中动态写值
