Cron表达式

在线Cron表达式生成器 - 码工具

用于控制任务触发时间

语法如下

6或7个域，表示不同的时间单位

域中可添加的特殊字符

  • * ：任意值（如 * 在“分”字段表示每分钟）

  • ? ：不指定（仅用于“日”或“星期”，避免冲突）

  • - ：范围（如 10-12 表示10点到12点）

  • , ：多个值（如 MON,WED,FRI 表示周一、三、五）

  • /：步长（如 0/15 表示从0开始每15秒）

  • L ：最后（如 L 在“日”字段表示月末最后一天）

  • W ：工作日（如 15W 表示离15号最近的工作日）

  • # ：第N个星期几（如 6#3 表示第三个周五）

‘星期’与’日‘一般只填一个，另一个使用？表示不指定，一起填可能会产生冲突

例：

0 0 12 * * ? 每天中午12点

0 15 10 ? * MON-FRI 工作日（周一到周五）上午10:15

0 0/5 14 * * ? 每天下午2点到2:55，每5分钟

0 0 22 L * ? 每月最后一天晚上10点

0 0 9 1W * ? 每月第一个工作日早上9点

0 30 10 ? * 6#3 每月第三个周五上午10:30

0 0 8-18/2 * * ? 每天8点到18点，每2小时

5 * * * * *    每分钟的第5秒

0/5 * * * * *    每5秒

与前端对比一下，可以发现有三个相似的方法，它们也称生命周期回调方法

  • @OnOpen → 连接建立回调

  • @OnMessage → 消息到达回调

  • @OnClose → 连接关闭回调

  • @OnError → 异常回调（可选）

这些方法不在自己代码中显示调用，是Servlet容器在相应事件发生时自动触发

端点？

关于websocket，我觉得还是挺有趣的，这里想细聊一下后端里websocket的作用？流程

首先，端点就是websocket中“可以连接”的东西，它包括了一个可连接的地址 + 一段处理代码，看上面

  • @ServerEndpoint(“/ws/{sid}”)：可链接地址，注意：这个注解用于将该类注册为WebSocket处理类，是Tomcat层（Servlet容器）的东西

  • 类中的方法：处理代码

前端与后端使用websocket通信，实际就是与后端的某个端点连接上，并间接(回调——特定事件触发调用)调用其中的方法，客户端连到哪个 URL，服务器就用哪个端点类来处理这条连接的全部生命周期

ServerEndpointExporter?

不知道你们会不会感觉奇怪，上面配置类注册的bean是什么？它好像返回的不是WebSocketServer这个端点类的实例，这挺怪的：它有什么用？

其实它是 WebSocketServer 这些端点类的“扫描仪”，用于识别项目中的端点类，注意，扫描的端点是给 Servelt 容器（tomcat）用的，而不是Springboot要用，对于普通的http接口，Spring 有一个DispatcherServlet 会拦截所有请求并分发给对应Controller，而 WebSocket 不走 DispatcherServlet ，它有专门的 WebSocket 升级过滤器/处理器，由 Servlet 容器直接管理，因此 Tomcat 在接收到 WebSocket 通信请求时，必须知道使用哪个端点处理。

在Springboot启动时，先加载所有bean，加载完成后，执行 ServerEndpointExporter 中的回调方法：将所有添加了ServerEndpoint注解的端点类的 Class 文件添加到 Tomcat 中，然后这个扫描仪的任务就结束了

后续就是建立连接时，Tomcat 按路径找到对应的 Class，如果发现这个 Class 在 Spring 中存在单例（bean），就会直接拿来用；如果没有，会用反射创一个新实例。这就是端点类上@Component注解的作用
