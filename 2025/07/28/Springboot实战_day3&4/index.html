复习：过滤器&拦截器

开始之前，我想复习一下登录校验的实现：两种方式

  • Filter：过滤器实现

  • Intercepter：拦截器实现

Fliter：

实现Filter接口，添加注解表示过滤器作用的访问路径，在doFilter中实现具体过滤逻辑，最后使用filterChain.doFilter放行到下一过滤器，否则直接return

Intercepter：

两个步骤：1、创建拦截器    2、注册拦截器

创建拦截器需实现HandlerIntercepter接口，实现preHandle方法，返回true时放行

创建配置类并实现接口WebMvcConfigurer，注册拦截器

但是项目中的配置类是这么写的

怎么不是实现WebMvcConfigurer接口？

一句话区别

WebMvcConfigurer是推荐方式，不会覆盖 SpringBoot 的自动配置
WebMvcConfigurationSupport是底层实现类，会禁用 SpringBoot 的自动配置

什么意思呢

  • WebMvcConfigurer
    是一个接口，只定义回调方法
    你实现它，SpringBoot 会自动识别并合并你的配置，不会破坏自动配置

  • WebMvcConfigurationSupport
    是一个类，提供了完整的 SpringMVC 配置。
    一旦你继承它，SpringBoot 的WebMvcAutoConfiguration就会失效，因为它会认为你要完全接管 MVC 配置

SpringMVC的自动配置中有什么

有静态资源映射，也就是当浏览器访问某个 URL 时，Spring 把请求直接转给项目里的静态文件（如 .html 、 .css 、 .js 、图片等），而不是交给 Controller 处理

项目中继承WebMvcConfigurationSupport，默认路径就失效了，需要手动配置静态资源映射

在配置类中找找看，还真有

这不是Swagger添加的配置吗，意思是当访问/doc.html时，Spring会到classpath:/META-INF/resources/ 目录下找doc.html文件并返回。既然要自定义静态资源映射，怪不得选择继承WebMvcConfigurationSupport

复习：AOP

面向切面编程

思考切面的作用流程：

找到一个切点，往里面切一个“面”：额外逻辑

因此AOP由两个关键部分组成：切点与面

切点

有两种描述切点的方式，形容一下它们的作用特点就是——

  • execution：描述切点的样子，所有符合描述的方法都将成为切入的目标

  • annotation：使用一个“切点标记”，添加这个标记的方法将成为切入的目标

例：execution——所有mapper包下的方法都是切入目标

annotation——添加“Autofill标记“的方法就是切入目标

AutoFill标记是什么？是一个自定义注解

逐行解释：

@Target(ElementType.METHOD)表示这个注解只能加在方法上

@Retention(RetentionPolicy.RUNTIME)表示这个注解在运行时保留，可以通过反射读取

public @interface AutoFill定义一个注解，名字叫 AutoFill

OperationType value();注解有一个必填参数，类型是 OperationType

上面这两种描述方式有一个名字。叫“切点表达式”

如果现在我告诉你，之后每一次想用切点的时候都得写这个表达式，你会怎么想

这也太长了

所以让我们给切点取个名——@PointCut的作用

现在，你可以使用AutoFillPoint来称呼这个切点了。当然，也可以不用

“面”

现在给出切点，我们要通过它们向程序中插入“面”，也就是代码

而众所周知，“面”是有两面的——正与反，因此”面“是有种类之分的

如果我们把原程序（长方体）拍扁，也看成一个面，那么我们现在要在这个程序的前还是后切入我们的”面“呢

前（Before），就是在程序执行前，先执行我们的”面“

后（After），就是在程序执行后，再执行我们的”面“

这就是“面”的作用时机，通常使用注解表示

例：

在原程序执行前切入”面“，SpringAOP在看见“面”含JoinPoint类型参数时，会将“切入信息”传进来

JoinPoint：连接点

切入信息指的是切入场景的信息，其中包括切入方法的信息等

JoinPoint
├── 被拦截的对象（target）
├── 参数数组（args）
├── 方法签名（Signature）
│ └── MethodSignature（向下转型后）
│ ├── getMethod() → 反射 Method
│ ├── getParameterTypes() → 参数类型数组
│ ├── getReturnType() → 返回值类型
│ └── getAnnotation(…) → 方法上的注解

什么是Signature？方法签名对象

Signature
“方法身份证”
只关心“被拦住的方法长什么样”（方法名、参数类型、返回类型、注解等）
MethodSignature
“身份证的详细版本”
因为拦截的是“方法”，所以 Spring AOP 给你一个专门描述方法的实现类

而这样的作用时机，有5种：

@Before：前置通知，在目标方法执行前执行

@After：后置通知，在目标方法执行后执行

@Around： 环绕通知，在目标方法执行前后都能执行

@AfterReturning：返回后通知，目标方法后执行，且有异常不执行

@AfterThrowing：异常后通知，仅在异常后执行

其实都只是在两个面上讨论罢了

最后，上述的“面”，实际上称为通知，现在可得等式

切面 = 切点 + 通知

获取到Class

三种方式

  • Class.forName()

  • 类.class()

  • 对象.getClass()

通过Class获取内部成员

Class类的方法有两个常见“关键词”

  • 关键词1：Declared

        带有这个词的方法只看本类自己声明的成员，不看父类，也不看 public 过滤

  • 关键词2：……s

        带有这个词的方法一般不用指定成员名，会返回一个含全部( )的数组，

通过方法获取到的内容有两种

  • 方法（构造方法、成员方法）

  • 变量

公共字段自动填充

很多数据库操作都有一些相同的填充字段，比如：更新人，更新时间

如果每次都要显示完成数据填充，那么会有很多冗余的代码……因此使用公共字段自动填充改进

操作步骤：

  • 自定义注解AutoFill，用于标记需要进行自动填充的方法

  • 自定义切面类AutoFillAspect，统一拦截添加了AutoFill的方法，并通过反射为公共字段赋值

  • 在Mapper中的目标方法上添加AutoFill注解

自定义注解

AutoFill

OperationType 是一个枚举类，规定了这个注解将要处理的两种类型

自定义切面类

不得不说，这一段要点挺多的

  • @Aspect 切面类

  • @Pointcut 声明切点

  • @Before 前置消息

  • 现在进入这个切面，有两种可能：update方法，insert方法。因为这两种方法对应的填充内容不同，所以要获取对应方法：joinPoint（连接点）.Signature（方法签名）.getMethod（获取方法）.getAnnotation（获取方法上的对应注解）若存在这个类型的注解，则返回它

  • autoFill是返回的注解，它含一个OperationType类型参数，我们通过.value()获取它

  • 通过joinPoint.getArgs获取方法的参数，我们规定第一个参数为操作对象

  • 准备赋值数据当前时间与操作人，操作人使用线程变量获取

  • 判断方法类型，对应两种填充逻辑：Update，Insert，使用反射通过entity对象拿到其中的方法，并设置值，需要通过try-catch捕获过程的异常，防止程序终止

”编译看左边，运行看右边“

声明类型只影响编译时期，包括强制类型转换也是，不会改变目标对象的实际类型，只是为了能调用这个对象的特有方法：如果左边是Object，调用Employee类的方法过不了编译。但实际运行时(看右边）它就是Employee类，是完全可以运行Employee的特有方法的

复习：Spring事物

@Transactional

把它贴在 类 或 方法 上，Spring 就会自动给该方法 织入事务边界——开始前开事务，成功后提交，异常后回滚

需在启动类上添加注解以开启注解方式的事物

@EnableTransactionManagement //开启注解方式的事务管理

上面Service层提到，菜品id是表主键（自增）,不能通过传参获取，为什么录入数据库后可以通过参数.getId获得id呢？是因为在Mybatis中使用了 useGeneratedKeys 与 keyProperty

由于前端要展示的数据分散在两个表中（种类在Category表），所以为前端返回VO类

概念：VO（View Object / Value Object）

“专门给前端（或接口调用方）看的 DTO”，里边只有 展示需要的字段，没有业务逻辑，也没有持久化注解

复习：pageHelper工具

使用方式

核心属性

  • pageNum int 当前页码

  • pageSize int 每页条数

  • total long 符合条件的总记录数

  • pages int 总页数

  • result / list List 当前页数据（与自身是 List 等价）

返回“分页统一返回对象”，包括记录数和具体内容

Mapper

有难度

主体部分：

查询dish表与category表中符合条件（动态）的数据，并展示dish表与categoryName

要点

  • , c.name as categoryName 取别名，由于在category表中它也叫name，这会导致这张表存在两个name字段，mybatis无法自动装配到类中，所以命名为类中成员变量名categoryName

  • left outer join 左外连接，拿左边表的全部记录，右边表能匹配就匹配，匹配不到用 NULL 补位

  • on 表匹配规则，要求dish的category_id字段等于category的id字段的两项才能匹配

  • name是传入对象的属性名

批量删除菜品

Controller

Service

主体逻辑：

  • 当前菜品是否能被删除——是否存在起售中菜品

    通过菜品id查询菜品：实现getById方法

  • 当前菜品是否能被删除——是否被套餐关联

    通过菜品id查询套餐id：实现getSetmealIdByDishId方法（或getSetmealIdsByDishIds）

  • 删除菜品表中的菜品数据

    通过菜品id删除菜品：实现deleteById方法

  • 删除菜品关联的口味数据

    通过菜品id删除口味数据：实现deleteByDishId方法

为什么这四个操作只有 通过菜品id查询套餐id 会使用批量查询方式？其他都是在外面套一层循环

首先它是查询操作，一次查多个遇到查不到的数据时不会出问题，但一次删多个遇到不存在的数据就会报错，导致整个sql语句中断，因此删除语句都逐个判断id是否存在后,再删除，保证数据正确的情况下是可以使用一条sql完成批量操作的

删除不存在项不会报错，后面重新设计时就使用了批量操作

其次是它的功能，这个操作的目的是查看表中是否存在任意符合规则的数据，有的话就报错，而通过菜品id查询菜品需要获取每个id的菜品数据，并查看其中的status属性是否为1（在售），这个额外的逻辑使得一次查询多个菜品这样的设计没什么意义
