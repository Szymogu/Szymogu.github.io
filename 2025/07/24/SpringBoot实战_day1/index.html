前端环境

直接启动项目提供的nginx即可

注意事项

1.路径不能有中文

2.启动失败有很有可能是端口号被占用，我选择修改配置文件中的端口号

nginx-1.20.2\conf\nginx.conf

当然也可以结束占用端口的进程，先使用指令查看端口是否被占用

  • -a ：显示所有选项，默认不显示LISTEN相关

  • -n ：以数字格式显示 IP 地址和端口号

  • -o ：显示每个连接的所属进程 ID

可能出现这样的结果

TCP 0.0.0.0:80 0.0.0.0:0 LISTENING 4

表明进程号为4的进程占用了这个端口，很有可能是一个system级别的服务，惹不起。。

所以最好的做法还是一直往后找到一个空端口，然后修改配置文件

关于nginx，还有两个重要的作用——反向代理&负载均衡

Nginx：反向代理

向后端服务器转发前端的请求

抓包前端的请求路径，发现与后端配置的路径是不一样的，但前端却可以请求成功，这就是因为nginx做了反向代理：修改前端请求路径

看一下nginx中是怎么配置的

nginx.conf

location模块用于规定对特定请求路径作用一套处理规则，其中proxy_pass指令用于指定转发到的后端服务器地址，此时nginx会把所有http://localhost:83/api/修改为http://localhost:8080/admin/，而api/后的动态资源路径会直接拼接到修改后的路径后面，即http://localhost:8080/admin/employee/login

刚好就是后端login请求处理类的路径

Nginx：负载均衡

既然nginx可以转发前端请求，那它能否选择要发给哪个后端服务器？当然可以，为什么要发给不同服务器？负载均衡——服务器集中的每台服务器都别闲着

看看nginx是中怎么配置的    

  • name：后端服务器组的名称，用于在proxy_pass中引用

  • server1,2：后端服务器地址

还可以选择一些均衡策略

  • 轮询：默认

  • weight：权重方式，默认为1，权重越高分配的请求越多

  • least_conn：最少连接方式，把请求优先分给连接数少的后端服务

比如这样，就是最少连接方式

还有一些可选项

  • 权重

请求分配的比例

  • 最大连接数

限制后端服务器的最大连接数

  • 健康检查

移除不可用的后端服务器

然而我没有这么多服务器，只有一个localhost，枯枯

后端环境

打开项目中的初始后端项目，包含比较完整的项目结构，但是基本上没有实现任何功能

尝试加载好父项目的pom文件，这一步巨看人品……我搞了一个下午

注意事项

1.可能要挂梯子？毕竟中央仓库在国外，有时候maven报错连接超时，多刷几遍应该也是可以的

2.注意项目使用的是jdk17

在idea中可以这样修改

  • 打开 “File” -> “Project Structure”

  • 在 “Project” 选项卡中，确保 “Project SDK” 指向 JDK 17

在配置一下外部项目仓库，使用gitee

登录 新建仓库 选择私有

创建完成后可以从项目找到仓库地址

数据库

使用项目提供的sql脚本直接创建数据库即可

我使用是datagrip用于可视化数据库，操作方式：右键连接–新建–来自路径的数据源–选择sql脚本

记得将数据库账号与密码配置到后端中

现在项目有一个小问题：数据库中的密码是明文存储的，这不安全，我们打算使用md5加密后存储。

找到项目的TODO，在校验密码前先对用户输入的密码做md5加密

DigestUtils 是 Apache Commons Codec 库中的一个工具类，用于生成各种摘要（哈希）值，例如 MD5、SHA-1、SHA-256 等。它提供了一种简单的方式来计算字符串或字节数组的摘要值

Swagger：常用注解

  • @ApiModel：用在类上，对数据模型类的说明

  • @APiModelProperty：用在属性上，描述属性信息
