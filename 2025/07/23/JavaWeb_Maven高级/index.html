Maven的更多版本控制技巧

分模块设计

在项目设计时期，将项目按功能拆分成若干个子模块。便于项目的管理维护、扩展，也方便模块的互相调用

如将项目组可能会使用的通用工具抽成一个模块，这样项目组中的多个项目想使用它时，都可以简单的导入这个模块，而不是一整个项目

注意，分模块是在项目设计的时候做的，一般不会说项目做到一半再分模块

继承与聚合

继承

父工程配置各个工程共有的依赖

简化依赖配置、统一管理依赖

思考如下场景，项目的工具模块和主要逻辑模块都需要使用lombok，难道我要在每个模块中单独配置lombok吗。当模块更多时，我们要配置多次，这很累

因此我们可以使用继承关系，让需要lombok的模块都继承一个父模块（父工程），由于Maven的依赖传递，它的子模块都会带上lombok

具体操作步骤：

1.创建maven模块parent，并设置打包方式为pom

什么是打包？

将项目的代码与资源等组织成可部署、可分发的格式

一般来说，java项目有三种打包方式

  • jar（Java Archive）：普通java包，将多个文件打包成一个单独文件，类似于压缩文件zip。jar也是Springboot项目的默认打包方式，jar包可以直接部署到tomcat服务器上，很方便

  • war（Web Application Archive）：Java Web 应用程序包，传统SpringMVC中用于打包JavaWeb应用程序的格式，它会将web程序的所有文件（包括前端项目）打包成一个文件，现在用的不多

  • pom（Project Object Model）：项目对象模型，父项目、聚合项目，Maven中用于管理子模块的模块打包方式，自身不包含任何代码（为什么这种打包方式要叫pom？可能是因为这类项目中起作用的就是pom文件吧）

idea创建新模块–选择java（不是maven archetype），create后在pom.xml中配置packagin属性为pom

版本锁定

父工程统一管理依赖版本

子工程引入依赖时，无需指定version，而是由父工程统一管理，维持整个项目依赖版本的统一性

思考如下场景，同样是lombok工具，三个子工程中有两个需要使用，如果直接配置在子工程中，某一次我想改变lombok的版本，我需要去项目中逐个查找并修改，但我现在将版本信息配置在父项目中，只用修改一次，所有子项目所用的均为父项目指定的版本

dependencies与dependencyManagement的区别

dependencies：直接依赖，父工程配置后，子工程会直接继承

dependencyManagement：仅管理依赖版本，不会直接依赖，若使用要在子工程中引入该依赖（无需指定版本）

聚合

快速构建项目

当我们打包一个项目时，maven会去本地仓库查找pom.xml文件中的依赖模块，若项目是分模块设计，并且之间存在依赖关系，要先把以来的模块先install（构建和安装项目到本地 Maven 仓库）好，最后才能打包成功

在尝试手动打包项目时，踩了两个小坑

1.java版本问题，子模块编译使用了java22，当前SpringBoot版本不支持

解决办法：先使用clean命令清理项目，再添加配置后，install即可

2.打包时会运行test中的代码，如果想不测试直接打包，可以点击“toggle Skip test mode”

聚合项目可以统一管理子模块的构建，通常就是父项目（这也是为什么父项目也称为聚合项目），当我们打包父项目时，聚合特性可以将多个模块组织成一个整体，自动构建好项目

使用方式

在父项目中配置需要同时构建的模块

此时会发现右侧的maven目录中，这两个子模块归到父模块下了，使用clean、package等命令时，会自动处理这些子模块

私服

架设在局域网内的仓库服务，用于代理位于外部的中央仓库。可上传jar包，也可下载jar包，可解决团队内部的资源共享与资源同步问题

一个公司可能会架设一台私服，用于传递公司内部的资源，同时这台私服会连接中央仓库，本地仓库没有这个依赖时，就会逐级向上查找，直到中央仓库，私服会将这个依赖缓存到本地

私服中的关键概念

  • 仓库（central）：从中央仓库下载的jar包、

  • release：发行版本资源

  • snapshot：快照版本资源（开发中）

可通过pom文件中的version属性指定发布仓库

携带SNAPSHOT：快照版本

  • 仓库组：多个仓库的集合，下载依赖时指定仓库组，即可在多个仓库中检索依赖

使用步骤：

1.在maven中设置私服的访问用户名&密码

settings.xml中的servers下

2.在maven中设置私服以来下载的仓库组地址

若配置了阿里云私服，需要替换掉

3.在profile中允许使用snapshot版本的依赖

4.在项目的pom文件中配置发布地址（仓库地址）

注意这里的id是在maven的settings中配置的，一个id会对应一套用户名与密码

最后使用deploy指令发布到私服
