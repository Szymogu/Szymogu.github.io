前置知识——一些系统调用

在做题的过程中，使用到的一些系统调用

fork()

这个在章节中讲过，用于创建子进程。它的返回值有三种

  • 小于0：请求失败

  • 等于0：我是子进程

  • 大于0：请求成功，这是您的子进程号

open()

用于创建一个新的文件描述符

“文件描述符”是一个用于表示已打开文件的整数值。在操作系统中，当程序打开一个文件时，操作系统会分配一个唯一的文件描述符给该文件，程序通过这个文件描述符来对文件进行读写等操作。

open系统调用会建立一个从文件到设备的访问路径，并为每个文件分配一个唯一的文件描述符，其他系统调用可以使用这个文件描述符操作对应的文件。文件描述符是唯一的，仅对调用open的进程有效（还有其子进程）。当调用open失败时，会返回-1，并设置全局变量errno来指明失败原因

“errno”是一个全局变量，用于存储错误信息。当系统调用或库函数执行失败时，程序会将一个特定的错误码赋值给errno，这个错误码可以用来指示失败的具体原因。

使用头文件以检查它的值#include<errno.h>

然后直接输出即可，默认值是0，表示没有错误

调用方式

path：将建立连接的文件或设备

oflags：访问模式（读写运行）当然，我会给出

  • O_RDONLY：以只读方式打开

  • O_WRONLY：以只写方式打开

  • O_RDWR ：以读写方式打开

　　另外，还有以下几种可选模式的组合（ 用按位或 || 来操作 ）：

  • O_APPEND：把写入数据追加在文件的末尾

  • O_TRUNC：把文件长度设置为零，丢弃已有的内容

  • O_CREAT：如果需要，就按照参数 mode 中给出的访问模式创建文件

  • O_EXCL：与 O_CREAT 一起使用，确保调用者创建出文件。使用这个模式可以防止两个程序同时创建同一个文件，如果文件已经存在，open 调用将失败

mode：使用O_CREAT标志创建文件时，需要给出创建文件的权限

S_IRUSR 文件属主可读 S_IRGRP 文件所在组可读 S_IROTH 其他用户可读
S_IWUSR 文件属主可写 S_IWGRP 文件所在组可写 S_IWOTH 其他用户可写
S_IXUSR 文件属主可执行 S_IWOTH 文件所在组可执行 S_IXOTH 其他用户可执行

write()

系统调用，将数据写入文件描述符（file descriptor）所指向的文件或设备

参数

fd：

文件描述符，表示要写入的目标文件或设备

标准文件描述符：

  • STDIN_FILENO（标准输入，文件描述符为 0）：通常与键盘输入关联
  • STDOUT_FILENO（标准输出，文件描述符为 1）：通常与终端或控制台关联
  • STDERR_FILENO（标准错误，文件描述符为 2）：通常与终端或控制台关联，用于输出错误信息

buf：

指向要写入的数据的缓冲区。buf 是一个指向 const void 的指针，表示数据的起始地址

count：

要写入的数据的字节数

返回值

成功：

返回实际写入的字节数

失败：

返回 -1，并设置全局变量 errno 以指示错误原因

错误码：

  • EAGAIN：资源暂时不可用（非阻塞模式下）
  • EBADF：文件描述符无效
  • EFAULT：缓冲区指针无效
  • EINTR：系统调用被信号中断
  • EIO：I/O 错误
  • ENOSPC：磁盘空间不足

close()

系统调用，用于关闭一个打开的文件描述符（file descriptor）

参数

fd：文件描述符，同样的，有这三个标准文件描述符：

  • STDIN_FILENO（标准输入，文件描述符为 0）：通常与键盘输入关联
  • STDOUT_FILENO（标准输出，文件描述符为 1）：通常与终端或控制台关联
  • STDERR_FILENO（标准错误，文件描述符为 2）：通常与终端或控制台关联，用于输出错误信息

返回值

成功：

返回 0，表示文件描述符已成功关闭。

失败：

返回 -1，并设置全局变量 errno 以指示错误原因

错误码：

  • EBADF：文件描述符无效
  • EINTR：系统调用被信号中断
  • EIO：I/O 错误

wait()/waitpid()

一般来说，参数填NULL，表示等所有子进程结束

  • -1：无子进程，此时会设置errno为ECHILD

  • 其他：已结束子进程的pid

waitpid()是wait()的变体，它允许父进程等待特定的子进程结束，或者在子进程结束时返回而不阻塞父进程（非阻塞模式）

pid：需等待的子进程号

status：子进程退出状态

options：控制 waitpid 的行为。例如，WNOHANG 表示非阻塞模式，WUNTRACED 表示等待被跟踪的子进程

同waitpid()，这个status指子进程退出状态。wait 写入status的子进程退出状态是一个整数，通常需要使用以下宏来解析：

  • **WIFEXITED(status)**：检查子进程是否正常退出。如果返回非零值，表示子进程正常退出

pipe()

用于进程间通信，在两个进程间建立一个管道，允许一个进程（称为“写端”）将数据写入管道，而另一个进程（称为“读端”）从管道中读取数据

pipe在调用进程的文件描述符表中创建两个文件描述符：一个用于写入（写端），一个用于读取（读端）

  • **pipefd[0]**：管道的读端文件描述符
  • **pipefd[1]**：管道的写端文件描述符

返回值：

  • 成功时返回 0
  • 失败时返回 -1，并设置全局变量 errno

写操作：

父进程或子进程可以通过管道的写端（pipefd[1]）向管道中写入数据：

  • fd：文件描述符，表示要写入的目标文件或设备。文件描述符通常通过 open、pipe、socket 等系统调用获得
  • buf：指向要写入的数据的指针
  • count：要写入的数据的字节数

返回值：

成功返回写入字节数，失败时返回-1

读操作：

父进程或子进程可以通过管道的读端（pipefd[0]）从管道中读取数据：

  • fd：文件描述符，表示要读取的源文件或设备
  • buf：指向存储读取数据的缓冲区的指针
  • count：缓冲区的大小，即最多读取的字节数

返回值

  • 成功时，返回实际读取的字节数，为0表示已全部读取

  • 失败时返回 -1并设置 errno

管道的读写操作是阻塞的，除非管道的另一端已经准备好

管道是一种简单的、单向的通信方式，可用于父子进程之间的数据交换

pipe()创建的管道称为匿名管道，这是由于它仅用于父子进程之间的通信。

有另一个系统调用mkfifo()，它创建的管道称为命名管道（FIFO）。这种管道可以通过文件系统中的一个特殊文件（FIFO 文件）进行访问，它允许不相关的进程进行通信。

  • pathname：FIFO 文件的路径名
  • mode：文件的权限模式，与 open 系统调用中的模式类似

既然这样，那很显然还有一种做法，就是使用共享内存，什么是共享内存呢？

它是一种高效的进程间通信（IPC）机制，允许多个进程访问同一块内存区域，从而实现数据的共享。因为数据不需要经过内核缓冲区的多次拷贝，它可以实现快速的进程间通信

关于共享内存，会用到的函数有：

  • ftok

  • shmget

  • shmat

  • shmdt

  • shmctl

理解了共享内存的原理，现在讲一下使用流程

首先根据唯一键值 创建一个共享内存段，但现在这个内存段还不能被进程访问，因为进程不知道他的位置，所以要映射共享内存，将共享内存的地址告诉进程，随后进程就可以随便访问这块区域了，一个进程不用再访问该内存时，要记得解除映射，当这个没有进程需要访问共享内存时，删除内存段

上面这个流程实际上有5步，它们已经被加粗了，这5步刚好对应上面五个函数

ftok()

ipc，进程间通信模块，需导入头#include<sys/ipc.h>

生成唯一键值，供后续创建或访问共享内存段使用

  • pathname：一个存在的文件路径名
  • proj_id：项目标识符，用于区分同一文件的不同用途

pathname很奇怪，函数要求一个文件路径，但实际上它真的只用到了“文件路径”，与文件里面写了什么没有一点关系。为什么这么设计？？

  • 唯一性

这里实际上用到了unix系统的特性，每个文件都在系统中有唯一的inode号，根据这个inode号能保证key的唯一性

  • 可预测性

使用同样的文件名，可以生成相同的key，进而找到同一块内存空间

  • 安全性

文件位置比随机数更难猜？我认为这里跟文件的访问权限关系更大，因为非法进程想要加入共享内存的访问，就要使用相同的key，也就是说调用者必须能访问到这个文件，只要规定好这个文件都有谁能访问，那么这块共享内存就会安全很多

对于使用文件的权限生成key，我有两个问题：

1.ftok()是否是系统调用？是否会有权限不够的情况？

我一开始以为ftok()是系统调用，可是系统调用的权限是很高的，系统去访问文件不应该会失败。结果是它并不是一个系统调用，是一个含系统调用的库函数，会因为权限不足导致一些调用失败，所以确实会有访问失败的问题

实际上，“系统去访问文件不应该会失败”这个理解是错的，ftok()中实现获取文件inode的关键系统调用是stat()，它会尝试访问那个路径，如果访问失败会返回-1，访问成功时返回一个文件信息结构体stat：

这时ftok()就可以根据这个结构体中的inode信息，结合projid生成一个独一无二的key了

2.文件设置了访问权限，当前进程没有访问权限，但知道文件的具体位置，能否得到 key 值？

不是用文件路径，而是要读取到文件的inode信息。inode需要文件系统告诉我们，那这中间就有权限检查，所以没有权限，知道位置也得不到key值

根据上面的信息，我做了一个小实验，通过设置文件的权限，看看是否能获取到key，首先我在同级创建文件forftok，000权限等级也可以访问到……所以我把这个文件塞入一个文件夹，给文件夹设置权限，结果发现这里使用到的不是rw，而是x——可执行权限，即设置100时，ftok可以得到key，而666不行

shmget()

shm…，共享内存模块，需导入头#include<sys/shm.h>

shared memory get

创建、获取共享内存段

  • key：由 ftok 生成的键值（IPC键值）；可使用 IPC_PRIVATE创建一个进程私有的共享内存段
  • size：创建共享内存段的大小（字节数），获取时忽视
  • shmflg：标志位，如 IPC_CREAT（若不存在则创建）、IPC_EXCL（若已存在返回错误）、权限标志（如0666表示读写权限）

返回值

  • 成功时返回共享内存段的标识符（shmid），一种IPC标识符
  • 失败时返回 -1，并设置错误码 errno

错误码

  • EACCES：访问权限不足
  • EEXIST：指定的共享内存段已存在（当使用 IPC_CREAT 和 IPC_EXCL 时）
  • EIDRM：指定的共享内存段已被删除
  • EINVAL：无效的参数（如 key 或 size）
  • ENOMEM：系统资源不足，无法创建新的共享内存段

它成功时返回一个共享内存段标识符，这个标识符怎么用呢？此时共享内存段并未映射到任何进程的地址空间，进程要想使用，要将共享内存段附加到进程中

这里提到的享内存段标识符是IPC标识符的一种，IPC标识符实际上对应唯一个IPC对象，系统为每一个IPC对象保存一个ipc_perm结构体，该结构说明了IPC对象的权限和所有者，并确定一个IPC操作是否可以访问该IPC对象

发现ipc对象这里同样有一个权限设置，可以表示共享内存段的权限，进一步提高了安全性

shmat()

shared memory Attach

共享内存附加到进程

  • shmid：由 shmget 返回的共享内存段标识符
  • shmaddr：期望的映射地址，通常设置为 NULL，让操作系统自动选择地址
  • shmflg：标志位，不设置为默认映射（读写）、SHM_RDONLY（只读映射）

它返回一个指向共享内存的指针(void*)，进程可以通过这个指针像访问普通内存一样访问共享内存中的数据

失败时返回(void*)-1,并设置错误码 errno

错误码

  • EACCES：访问权限不足
  • EIDRM：指定的共享内存段已被删除
  • EINVAL：无效的 shmid 或 shmaddr 参数
  • ENOMEM：系统资源不足，无法完成附加操作
  • EAGAIN：如果指定了 IPC_NOWAIT 标志，且当前没有可用的共享内存段

shmdt()

shared memory Detach

解除进程与共享内存的映射

  • shmaddr：通过 shmat 返回的共享内存地址

返回值

  • 成功：返回 0
  • 失败：返回 -1，并设置 errno 以指示错误原因

错误码

  • EINVAL：指定的地址无效，或者该地址不是当前进程附加的共享内存段的起始地址

  • ENOMSG：指定的地址不是当前进程附加的共享内存段的起始地址

  • EFAULT：指定的地址无效，或者该地址不在当前进程的地址空间中

shmctl()

shared memory Control

用于控制共享内存段，有以下操作

  • IPC_RMID：删除共享内存段，不需要额外的状态信息，buf 传入 NULL
  • IPC_SET：设置共享内存段的权限和状态，需要传入一个指向 shmid_ds 结构的指针
  • IPC_STAT：获取共享内存段的状态信息，需要传入一个指向 shmid_ds 结构的指针
  • shmid：共享内存段标识符
  • cmd：命令，如 IPC_RMID（删除共享内存段）
  • buf：一些操作要提供，共享内存段状态结构体指针

shmid_ds：共享内存数据结构（Shared Memory Data Structure）

返回值

  • 成功：返回 0。
  • 失败：返回 -1，并设置 errno 以指示具体的错误原因

错误码

  • EACCES 没有足够的权限执行指定的操作（如 IPC_SET 或 IPC_RMID）

  • EFAULT 参数 buf 指向无效的内存地址（当 cmd 为 IPC_SET 或 IPC_STAT 时）

  • EIDRM 指定的共享内存段已被删除

  • EINVAL 参数 shmid 或 cmd 无效

  • EPERM 没有足够的权限执行操作（如 IPC_SET 或 IPC_RMID）

  • ENOSYS 系统不支持该操作

上面提到，shmid使用的key除了由ftok生成，还可以直接填IPC_PRIVATE,表示创建一个仅本进程可访问的共享内存段，我们试着用一下

设置进程优先级

which：指定优先级的类型，可以是以下值之一：

  • PRIO_PROCESS：设置单个进程的优先级
  • PRIO_PGRP：设置进程组的优先级
  • PRIO_USER：设置用户的所有进程的优先级

who：指定要调整优先级的进程或进程组的 ID。如果 which 是 PRIO_USER，则 who 是用户 ID；如果是 PRIO_PROCESS 或 PRIO_PGRP，则 who 是进程 ID 或进程组 ID（嗯，猜猜看当这里代表进程号时，填0是什么意思。没别的，因为实际进程号不存在0，但是再想想fork调用的返回值……有点意思）

value：新的优先级值。优先级的范围通常是 -20 到 19，其中 -20 是最高优先级，19 是最低优先级

注意，普通用户只能降低优先级（++），提高优先级（–）要root

查看进程优先级

which：指定要获取优先级的对象类型，可以是以下值之一：

  • PRIO_PROCESS：指定单个进程
  • PRIO_PGRP：指定进程组
  • PRIO_USER：指定用户的所有进程

who：指定要获取优先级的具体对象，其含义取决于 which 参数：

  • 如果 which 是 PRIO_PROCESS，who 是进程 ID
  • 如果 which 是 PRIO_PGRP，who 是进程组 ID
  • 如果 which 是 PRIO_USER，who 是用户 ID
  • 如果 who 是 0，则表示当前进程（你猜对了吗）、当前进程组或当前用户的所有进程（根据 which 的值）
